

														Kubernetes
	
	Kubernetes ek open-source platform hai jo Containezied Application ke deployment ko scale and manage or automated karne me use hota hai.
		jo Application ko downtime jane se bhi bachata hai.
		
--->Google ne devlope kiya tha aur ab cloud Native Compiting Foundation kr under maintain hai.


Featurs:====>
				1.Scalabillity
				2.High-Availablilty
				3.Security
				4.Integration With AWS
				
Kubernetes Key Concept:-->
							POD, NODE, CLUSTER, SERVICES, DEPLOYMENT
							
							POD:---> Ye continer ka group hota hai jo ek sharde enviroment me run karta hai.
									**ye K8s ki smallest unit hoti hai **
							NODE:--> Ye Physical ya virtual machine hai jo kubernetes cluster ka hisa hota hai.
									Har ek node me kubelet hota hai jo container ki manage karta hai.
							CLUSTER:-->	Multipul nodes ke group hota hai. Jisme kubernetes deploy hota hai.
										ek master node hota hai, jise  Control plane kahte hai . OR worker-node hote hai jen me Application deploy hoti hai pod ke ander Container ke ander run hoti hai.
							SERVICES:-->	Ye abstract layer hoti hai , jo ek stable endpoint provide karati hai.
											Taaki pods ko access kar sake or Application ko access kar sake. 
											***Cheye pod ka ip change hota rhe.****
							DEPLOYMENT:->	Deployment me application ki configuration hoti hai jo batata hai ki application ko Kubernetes me kaise deploy, update, aur manage karna chahiye.

							1. kube-api Server:---> Kubernetes me Cluster ke ander control plane hota hai agar kubernetes ke cluster me kisi bhi resources ko manage karna hota hai to kubernetes API-Server ke through he intract kar skte hai.
													Kube-api Server, Most importent in k8S Cluster
													API Server hi Kubernetes ka main entry point hai, har request yahin aati hai.
													 All Cluster Request heandle Kube-api server .
													 -->	Request chaye user ya cluster  through ke or intranl je  through aaye sabhi request ko heandle karta hai kube-api server .
													 -->	RESTFUL arcitecture ka use karta hai.
													 -->	Cluster ke resource ko state reteive karna ya Modify karne ke liye  HTTP request accpect karta hai kube-api server .
									
									API-OBJECTS:---> kubernetes main object ko define karna hota hai jo desired state ko represent karta hai . Yeh object YAML/JSON Formate me hota hai.
													 POD:--> 		apiversion:v1 , 	kind:-> pod
													 Deployment:--> apiversion:apps/v1  Kind:--> Deployment
													***In Object ke through hi pod Create , delete , scale kaiya jo skata hia.*****
									**** API se interact karne ke liye kubectl hota hai  jo internally api server ko request send karta hai.****
									
							
							2. Kube Controller manage:--->	Kube Controller like a Brain hai jo cluster ke self-healing capability ka important hissa hai.
															Kube Controller Manager continuously etcd ko monitor karta hai.
															Kubernetes me Sabhi Resources ko ,manage kata hai jese NameSpace, Node Controller , Deployment Controller , Replicaset Controller ,Service Account Controller ,etc.
															use:-	
																1.	Jab bhi user koi deployment karta hia .
																2.	API Server request ko accept karta hai aur etcd me store karta hai (as desired state).
																3. 	Kube Controller Manager desired state ko etcd se read karta hai.
																4.	Wo actual state ko compare karta hai:
																5.	Agar 3 pod chahiye, aur sirf 2 pod chal rahe hain → 1 pod aur create karega.
																6.	Ye actions kube-apiserver ke through perform karta hai.
													================================================================================						
													|	1. User																		|
													|	   ↓																		|	
													|	   Runs: kubectl apply -f deployment.yaml									|
													|																				|
													|	2. Kube API Server															|
													|	   ↓																		|
													|	   - Accepts & validates request											|
													|	   - Stores object (Deployment, Pod, etc.) in etcd (→ desired state)		|
													|																				|
													|	3. etcd																		|
													|	   ↓																		|
													|	   - Cluster ka brain (database)											|
													|	   - Deployment object now saved here										|
													|																				|
													|	4. Kube Controller Manager (→ Deployment Controller)						|
													|	   ↓																		|
													|	   - Watches etcd via API Server											|
													|	   - Detects: “Naya Deployment create hua hai, 3 pods chahiye”				|
													|	   - Creates a ReplicaSet (also stored in etcd)								|
													|																				|
													|	5. ReplicaSet Controller													|
													|	   ↓																		|
													|	   - Detects ReplicaSet object from etcd									|
													|	   - Launches required pods (via API Server)								|
													|																				|
													|	6. Kube Scheduler															|
													|	   ↓																		|
													|	   - Watches API Server for unscheduled Pods								|
													|	   - Filters & scores nodes													|	
													|	   - Assigns pod to best node (binding step)								|
													|																				|
													|	7. API Server (bining info update)											|
													|	   ↓																		|
													|	   - Updates pod's nodeName													|
													|	   - Notifies target node’s Kubelet											|
													|																				|
													|	8. Kubelet (on selected worker node)										|
													|	   ↓																		|
													|	   - Pulls container image													|
													|	   - Creates container via container runtime (Docker/CRI-O)					|
													|	   - Runs the pod															|
													|																				|
													|	9. Pod Running																|
													|	   ✅ DONE!																	|
													=================================================================================
													
															 
														1. Node Controller:---> 		→ Nodes ka health check karta hai (heartbeat dekh kar)
																						→ Agar Node down ho gaya, to pods reschedule karta hai
																						
														2. Replication Controller:-->	→ Ensure karta hai ki specified number of pod replicas always chal rahe ho
																						→ Agar 1 pod crash ho jaye, to naya pod launch karta hai
														
														3. Deployment Controller:--->	→ Deployment ke rolling update, rollout, rollback manage karta hai
																						→ ReplicaSet ke through pods ka version control karta hai
																						
														4. ReplicaSet Controller:---->	→ ReplicaSet object ke pods ka count maintain karta hai
																						→ Ye Deployment Controller ke under work karta hai.
																						
														5. Job Controller:------>		→ One-time tasks (batch jobs) run karne ke liye pod manage karta hai
																						→ Job complete hone par pod ko clean karta hai.
																						
														6. CronJob Controller:------>	→ Time-based (scheduled) jobs run karta hai
																						→ Jaise: “Run backup daily at 2 AM”.
																						
														7. Service Account Controller:->	→ Har naye namespace me automatic service account banata hai
																							→ RBAC ke liye jaruri hota hai.
	
														8. Namespace Controller:---->	→ Jab koi namespace delete hota hai, to uske saare resources ko cleanup karta hai.	
															
														9. ResourceQuota Controller:->	→ Namespace ke andar CPU, memory, pod count ki limits enforce karta hai.

														10. Horizontal Pod Autoscaler (HPA) Controller:-->	→ CPU/Memory usage dekh kar pods ka count badhata ya ghatata hai.


															
															
							3. kube-scheduler:-----> 	kube-scheduler Kubernetes ka ek core component hai.
														kubernetes me kube-scheduler ka kaam hota hai ki kon se worker node par pods ko schedule karna hota hai jo best fit node par deploy krta hai.
														Jab koi naya Pod create hota hai (Deployment, Job, CronJob ke through), to initially us pod ka node assign nahi hota. Kube-scheduler ka kaam hai us pod ke liye best-suited node select karna.
														Phale kube-scheduler check karta hia ki kon se node ki health fit hai dir pod ko deploy karta hai.
													
													_____________________________________________________________________________
													|How to Work Kube-schedule:-												|
													|																			|	
													|	1. Pod Create Hota Hai (Deployment, Job, etc.)							|
													|	   ↓																	|	
													|	2. Pod initially `Pending` hota hai										|
													|	   (nodeName = null)													|
													|	   ↓																	|
													|	3. Kube Scheduler detect karta hai pending Pod							|
													|	   ↓																	|
													|	4. Ye sari Nodes ko evaluate karta hai:									|
													|	   - Filter: Jo match nahi ho rahi unhe hata deta hai					|	
													|	   - Score: Bachi hui nodes ko score deta hai							|
													|	   ↓																	|
													|	5. Sabse best-scored node select karta hai								|
													|	   ↓																	|
													|	6. Scheduler API Server ko batata hai:									|
													|	→ "Is pod ko node-2 pe bhejo"											|
													|	   ↓																	|
													|	7. API Server pod me `nodeName` update karta hai						|
													|	   ↓																	|
													|	8. Us node ka `kubelet` us pod ko run karta hai							|

							4. ETCD:--------------->	🧠 ETCD Kya Hota Hai:-	Full Cluster ki information ETCD me store hoti hai . etcd ek distributed key-value store hai
																				Jo Kubernetes ke poore cluster ki state ko save karta hai.
														_____________________________________________________________________________________________________________
														⚙️ How to Work ETCD :-	
																				1. API Server → etcd me data write karta hai (create/update resources)
																				2. Controllers & Scheduler → etcd se data read karte hain
																				3. etcd data store karta hai as key-value pairs
																				4. Ye strongly consistent & highly available hota hai (Raft algorithm use karta hai)
														_____________________________________________________________________________________________________________
														
														📦 ETCD Kis Kaam Aata Hai:- 
																				Pod, Deployment, ConfigMap, Secret, Service — sabka desired state yahan store hoti hai
																				Cluster Nodes, RBAC Rules, Events — sabka metadata
																				Agar etcd delete ho jaye → cluster ki memory chali jaati hai
														_____________________________________________________________________________________________________________
														📁 ETCD Me Kya-Kya Store Hota Hai:-->
																							Resource Type			Example
																							Pods					kube-system/nginx-pod
																							Deployments				apps/my-backend-deploy
																							ConfigMaps				default/app-config
																							Secrets					kube-system/token
																							Nodes	  				worker-1, worker-2
																							ServiceAccount	 		default
																							RBAC Rules				ClusterRoleBindings
														_______________________________________________________________________________________________________________
														🔙 ETCD Backup Kaise Lete Hain:- First create Directory mkdir -p /backup
																							change Permission and chown Permission 700 and owner root:root
																					Backup:- 
																					
																						export ETCDCTL_API=3
																						etcdctl snapshot save /backup/etcd-snapshot.db \
																						--endpoints=https://127.0.0.1:2379 \
																						--cacert=/etc/kubernetes/pki/etcd/ca.crt \
																						--cert=/etc/kubernetes/pki/etcd/server.crt \
																						--key=/etc/kubernetes/pki/etcd/server.key

														🔁 ETCD Restore Kaise Karte Hain:-
																					Etcd cluster down ho gaya (crash)
																					Naye node me restore:-
																											etcdctl snapshot restore /backup/etcd-snapshot.db \
																											--data-dir /var/lib/etcd-restore

			✅ Kubernetes Application Lifecycle Management (ALM):-->
																	🔁 ALM ke Stages:
																						Stage				Description
																						1 Deploy			Pod, Deployment, Service create karna
																						2️ Update			RollingUpdate se app ko upgrade karna
																						3️ Scale				Manual ya HPA se replicas badhana/ghatana
																						4️ Health Check		Liveness & readiness probes se monitoring
																						5️ Restart/Repair	CrashLoop se recover karna
																						6️ Rollback			Galat deployment ke baad pichhle version pe jana
																						7️ Delete			Resources ko clean tareeke se hata dena
			🔧 Kubernetes Tools for Lifecycle Management:--->
																	Tool / Resource						Kaam
																	Deployment							Version control, rolling update
																	ReplicaSet							Pod scaling & health
																	StatefulSet							Stateful apps ke liye lifecycle
																	Job / CronJob						One-time ya scheduled jobs
																	HPA (Horizontal Pod Autoscaler)		Load ke hisaab se scaling
																	ConfigMap / Secret					Runtime config changes
																	Probes (liveness/readiness)			App ki health monitor
																	kubectl rollout						Update & rollback control
																	Helm								Complete ALM via charts
										____________________________________________________________________________________________________________________________________________										
										| Lifecycle Flow:--->																														|
										|						1. Dev pushes code to GitHub																						|
										|								↓																											|
										|						2. CI builds docker image																							|
										|								↓																											|
										|						3. Image pushed to registry (ECR/DockerHub)																			|
										|								↓																											|
										|						4. Deployment YAML apply via kubectl or Helm																		|
										|								↓																											|
										|						5. Kubelet pulls image via containerd																				|
										|								↓																											|
										|						6. App runs in Pod, readiness probe OK																				|
										|								↓																											|
										|						7. Traffic routed via Service																						|
										|								↓																											|
										|						8. Update code → new image → rolling update																			|
										|								↓																											|
										|						9. Issue found? Rollback with:																						|
										|						   kubectl rollout undo deployment/my-app																			|
										|								↓																											|
										|						10. Delete: kubectl delete deployment my-app																		|

				HPA ( Horizontal Pod AutoScaling ______________________________________________________________________
								Kubernetes me HPA (Horizontal Pod Autoscaler) ek controller hai jo pod ki count ko automatically adjust karta hai workload ke load ke hisaab se.
								Agar load zyada ho → HPA naye pods launch

								HPA continuously metrics server se pod ka CPU / Memory usage (ya custom metrics) monitor karta hai.
								Target utilization set hota hai (jaise 50% CPU).
								Agar average usage target se zyada hai → pods ka count badhata hai.
								Agar average usage target se kam hai → pods ka count kam karta hai.
								Ye scaling Deployment, ReplicaSet, ya StatefulSet par hoti hai.
								####################################################################################################
								
								1. Manual:-

																PV and PVC
															==================
															
	PV (PersistentVolume):-
							PV ek cluster-level storage resource hota hai.
							Ye ek actual storage location hai jo Kubernetes cluster me available hota hai (jaise NFS, AWS EBS, Azure Disk, Local Disk).
							PV ka lifecycle Pod se independent hota hai — Pod delete hone par PV apna data rakhta hai.
							
	PVC (PersistentVolumeClaim):-
								PVC ek request hota hai storage ke liye.
								Tumhara application directly PV nahi use karta — wo PVC banata hai, aur PVC automatically matchingPV se bind ho jata hai.
								PVC me tum specify karte ho:-
																Size (e.g., 10Gi)
																Access Mode (e.g., ReadWriteOnce, ReadOnlyMany, ReadWriteMany)
																StorageClass (optional)
								
								
		Real-world example:-
							📦 Scenario: Tumhare paas ek e-commerce app hai jo customer ke order data ko database me store karta hai.
							Agar tum DB ka pod delete karke naya pod create karoge, to normal case me sab data lost ho jayega kyunki pods ephemeral hote hain.
							Is problem ka solution → PV & PVC.
							Step-by-step:
								1. Admin ek PV banata hai jo cluster ke storage (e.g., AWS EBS volume, NFS share, ya local path) ko point karta hai.
								2. Developer PVC banata hai, bolta hai "Mujhe 20Gi ka storage chahiye ReadWriteOnce mode me".
								3. PVC automatically PV se bind ho jata hai.
								4. Pod ke volumeMounts me PVC attach kar dete ho.
								5. Ab Pod delete bhi ho jaye, data PV me safe rehta hai — naye Pod ko same PVC mount kar do, data wapas mil jayega. ✅


		Real-life benefits:-
							Data persistence: Pod delete hone par data safe rehta hai
							Portability: PVC se tum storage easily alag-alag Pods me reuse kar sakte ho
							Cloud ready: AWS EBS, GCP Persistent Disk, Azure Disk, NFS — sab integrate ho jata hai
							Separation of concerns: Admin storage manage karta hai, developer bas PVC ke through use karta hai

												Probes
											--------------
											
											
		pod/container ke health aur availability ko check karne ke liye use hote hain.
		Probe basically ek health check mechanism hai jo kubelet (node ka agent) ke through chalta hai.
		Har probe ek handler hota hai jo check karta hai ki container healthy hai ya nahi.
		
		Kubernetes me 3 types ke probes hote hain:-
		1. Liveness Probe 🟢:-
								Ye check karta hai ki container abhi tak alive (running) hai ya nahi.
								Agar liveness probe fail hota hai → kubelet container ko restart kar deta hai.
		
		2. Readiness Probe 🟡:-
								Ye check karta hai ki container traffic serve karne ke liye ready hai ya nahi.
								Agar readiness probe fail hota hai → container ko Service se temporarily remove kar diya jata hai (traffic nahi milega).
								
		3. Startup Probe 🔵:-
								Ye check karta hai ki container sahi se start hua hai ya nahi.
								Agar startup probe fail hota hai → container ko restart kar diya jata hai.
								Ye special tab useful hai jab app ko bohot time lagta hai start hone me.
								Jab startup probe set hota hai, tab tak liveness/readiness probe run nahi hote.
								
		🔹 Difference Table:-
		
			Probe Type			Purpose												Agar Fail Ho Jaye To?
			Liveness			Container alive hai ya nahi							Container restart hoga
			Readiness			Traffic serve karne ke liye ready hai ya nahi		Pod ko service se hata diya jata hai
			Startup				App sahi se start ho payi ya nahi					Container restart hoga	
			
			
		👉 Summary in short:
							Liveness Probe → Restart karna hai ya nahi.
							Readiness Probe → Traffic dena hai ya nahi.
							Startup Probe → App ready hone ka initial wait.

								Kubernetes Sidecar aur Init Container
							---------------------------------------------
		
	👉	Init Container: “Setup worker” → Sirf start hone pe kaam karega, phir exit.
		Sidecar Container: “Assistant worker” → Saath-saath main app ke support ke liye hamesha chalega.

		
	🔹 1. Init Containers:-
							Init containers pod ke start hone se pehle run hote hain. Ye containers ek type ke helper hote hain jo main 
							application container ke run hone se pehle kuch kaam karte hain.
							
						1.	Init container hamesha pehle complete hota hai, uske baad hi main container start hota hai.
						2.	Ek pod mein multiple init containers ho sakte hain, aur wo sequentially (ek ke baad ek) chalenge.
						
						
						use cases:-
						1.	Database ka wait karna jab tak DB ready ho.
						2.	Config files generate karke volume mein daalna.
						3.	Permission set karna kisi folder pe.
						5.	External services ke health-check karna.
						
						
						
	🔹 2. Sidecar Containers :-
							Sidecar container ek pod ke andar dusra container hota hai jo main container ke saath parallel me chalta hai.
							Ye main container ko supporting functionality provide karta hai.
						
						(use cases) :-
						1.	Logging sidecar: Logs ko ek container se collect karke external system ko bhejna.
						2.	Monitoring/metrics exporter.
						3.	Proxy (e.g. Envoy, Istio sidecar for service mesh).
						4.	File sync karna ya config reload karna.
						
	Sidecar Container main use case login and monitoring karne me aata hai.
